# Digest 对话收纳

将当前对话中的有价值信息提取并归入系统的持久化层。

## 触发

- `/digest` 或 `/digest 只记项目信息`
- 自然语言："总结一下这次对话"、"归入系统"、"收纳一下"、"记住我们聊的"

## 工作流

### Step 1：扫描当前对话，提取五类信息

| 类型 | 识别信号 | 目标位置 |
|------|----------|----------|
| 项目上下文 | 新项目、阶段变化、产品定义、目标人群、关键决策 | `5-Identity/CONTEXT.md` |
| 新观察到的偏好（未确认） | 本次对话首次出现的工作习惯、交互偏好、品味倾向 | `6-System/working-memory/candidates.md`（暂存） |
| 已确认的稳定偏好 | 通过本次 /digest 用户明确确认的候选条目 | `5-Identity/PROFILE.md` 或 `6-System/working-memory/OPERATING_RULES.md` |
| 可复用知识/模式 | 方法论、框架、模板、成功的任务执行模式 | PARA 对应位置 或 `6-System/patterns/` |
| Identity 相关 | 使命/目标/信念/策略变化 | `6-System/pending_approvals.md`（B类审批） |

### Step 2：读取全部目标文件，逐条对比

**必须在展示前完成。** 读取所有可能写入的目标文件，将 Step 1 提取的每条信息与已有内容对比，标记状态。

#### 四种状态

| 状态 | 含义 | 处理 |
|------|------|------|
| 🆕 新增 | 目标文件中无相关内容 | 正常展示，追加写入 |
| 🔄 更新 | 目标文件中有同主题旧版本，本次有增量 | 展示新旧对比，替换旧内容 |
| ⚠️ 矛盾 | 与已有内容存在方向性冲突 | 高亮冲突点，必须用户裁决 |
| ⏭️ 跳过 | 与已有内容实质重复，无增量 | 静默丢弃，不展示 |

#### 各目标的对比策略

**CONTEXT.md — 同主题更新制：**
- 按 `## 活跃项目` / `## 近期关注` / `## 近期决策` / `## 工作模式偏好` 分区扫描
- 同一个项目名命中 → 逐字段比较（阶段、定义、受众、关键决策）→ 有增量标🔄（只写增量），无增量标⏭️
- 新决策与已有决策方向相反 → 标⚠️，展示「旧：X → 新：Y」让用户裁决
- 项目已不活跃 → 标🔄，建议移到归档或删除

**PROFILE.md — 语义冲突检测：**
- 按分类区块匹配（AI交互偏好、写作与表达偏好、任务执行偏好、学习理解偏好）
- 新条目与已有条目含义相同或被已有条目包含 → ⏭️
- 新条目与已有条目方向冲突（例：已有"简洁直接"，新提取"喜欢详细展开"）→ ⚠️，给出场景化建议（"A情境-X，B情境-Y"）
- 新条目是已有条目的细化/补充 → 🔄，建议合并为一条

**candidates.md — 双重去重：**
- 新观察与 candidates.md 已有候选含义重叠 → ⏭️
- 新观察与 PROFILE.md 已晋升条目含义重叠 → ⏭️（已被吸收，不再暂存）
- 新观察与 OPERATING_RULES.md 已有规则重叠 → ⏭️

**OPERATING_RULES.md — 规则冲突检测：**
- 新规则与已有规则覆盖同一操作场景 → 🔄（合并）或 ⚠️（矛盾时用户裁决）
- 新规则与已有规则无关 → 🆕

**patterns/ — 模式重复检测：**
- 读取 `6-System/patterns/` 下所有 `.md` 文件的 summary/标题
- 新模式核心方法论与已有文件重叠 → 🔄（合并到已有文件）
- 新模式独立 → 🆕

### Step 3：读取 candidates.md，处理待确认候选

读取 `6-System/working-memory/candidates.md`，展示所有待确认候选条目。

**展示前同样执行对比**：每条候选与 PROFILE.md / OPERATING_RULES.md 已有内容对比，如果已有内容已覆盖该候选，标注⏭️并建议直接清理。

```
candidates.md 中有 N 条待确认偏好：
- [2026-02-25] 观察：偏好用类比理解复杂概念 | 来源：记忆系统设计讨论
  → ⏭️ PROFILE.md 已有"倾向用类比理解复杂概念"，建议清理
- [2026-02-25] 观察：XXX | 来源：YYY
  → 🆕 确认晋升到 PROFILE.md？[是/否/场景化]
```

### Step 4：合并展示本次结果

只展示 🆕/🔄/⚠️ 三种状态的条目（⏭️ 已被静默丢弃）。

格式示例：
```
本次对话提取（已对比去重）：

[项目上下文] → CONTEXT.md
- 🔄 外挂大脑系统：阶段更新「阶段四就位 → 记忆系统改造完成，进入日常使用」
  旧：完成记忆系统改造（阶段零～阶段四+L1全部就位）
  新：记忆系统改造完成，进入日常运维模式

[新偏好候选] → candidates.md
- 🆕 从产品/体验角度理解技术决策的倾向

[可复用模式] → 6-System/patterns/
- 🆕 系统架构设计方法：先诊断问题 → 决策文档 → 分阶段执行

[矛盾项 - 需裁决]
- ⚠️ CONTEXT.md 近期决策：旧「暂不做 Google Ads」←→ 新「启动 Google Ads 测试」
  → 请选择：[用新替旧 / 保留旧 / 两条都留并注明时间]

确认写入？（矛盾项需先裁决）
```

### Step 5：用户确认后执行

按确认范围写入：

**🆕 新增**：追加到目标位置

**🔄 更新**：找到目标文件中的旧内容，原地替换（不是追加）

**⚠️ 矛盾**：按用户裁决结果处理：
- 用新替旧 → 原地替换
- 保留旧 → 不写入新内容
- 两条都留 → 标注时间线并列保留
- 场景化 → 合并为「A场景-X，B场景-Y」格式

**写入规则（不变）：**

- 更新 CONTEXT.md：A类变更，直接写入
- 写入 candidates.md（新观察的偏好）：追加格式 `- [日期] 观察：[内容] | 来源：[对话主题]`，A类
- 晋升到 PROFILE.md（candidates.md 确认的条目）：写入对应分类下，A类
- 写入 patterns/：文件命名 `[任务类型]-[具体方法].md`，按模板格式写入
- 创建新笔记（可复用知识）：按 intake 流程处理
- Identity 提案：写入 `6-System/pending_approvals.md`，包含依据和回滚方案

### Step 6：报告变更

列出所有实际写入，按状态标注：
```
已完成：
- 🔄 CONTEXT.md：更新外挂大脑系统阶段描述（替换旧内容）
- 🆕 candidates.md：新增 1 条偏好候选
- ⏭️ 跳过 2 条（与已有内容重复）
- ⚠️ 1 条矛盾已按用户裁决处理（Google Ads 决策：用新替旧）
```

## CONTEXT.md 写入规范

项目条目支持子信息展开：

```markdown
## 活跃项目
- MyWear：可穿戴产品，进入营销阶段
  - 定义：AI 驱动的个性化穿搭推荐
  - 受众：北美 25-35 岁健身人群
  - 阶段：产品定义完成 → 营销冷启动
  - 关键决策：先做 TikTok + Instagram，暂不做 Google Ads
```

不是所有项目都需要子信息。一行能说清的就一行。

## 边界

- 不捏造对话中没有的信息
- 不自动执行 Identity 变更
- 新观察到的偏好先写 candidates.md，不直接写 PROFILE.md（要经用户确认）
- 如果对话纯闲聊没有可提取信息，直接说"本次对话无需归档"
- **对比是强制步骤**：展示提取结果前，必须先完成 Step 2 的全量对比。跳过对比直接展示是违规操作
