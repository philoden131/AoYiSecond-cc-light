# 任务目标：根据本工程化代码文件从**产品视角**倒推出一套“长期可维护、可讲清楚产品在做什么”的文档系统。代码是参考证据，不是叙事主体。文档服务对象是产品、设计、运营、未来的新成员，而不是工程实现。    
# 核心原则：  
以「用户能感知到的行为」作为第一性事实，而不是以代码结构作为第一性事实。  
代码只用于校验和补全，不主导结构。

0）对 AI 的角色设定（非常重要）   
你是产品文档构建者  
你只关心用户看到的能力、路径和结果  
除非必要，不描述实现细节

当你引用代码，只用于确认页面存在、能力存在、约束存在，不解释怎么实现。

1）先搭建“产品文档世界观”，不写任何细节  
创建一个 docs/product/ 目录，并声明这是唯一主入口。

在 product/README.md 中只回答三件事：  
这个产品解决什么问题  
核心使用对象是谁  
产品的能力边界在哪里

这一步禁止出现页面、按钮、模型、接口，只写产品语言。

2）用“用户完成目标”来倒推，而不是用页面  
做的第一件事不是扫页面，而是回答：  
一个用户来 MyWear，最终能完成哪些事情？

要求 AI 输出一份「用户目标清单」，例如：  
建立自己的衣橱  
快速录入衣物  
获得可执行的穿搭建议  
保存、复用或分享结果

每一个目标一句话，不超过两行，不涉及任何 UI。

这一步的产物是 docs/product/goals.md  
这是整套文档的锚点。

3）从“目标”反推“能力”，而不是反推代码  
 针对每一个用户目标，拆出必须存在的产品能力。

能力的定义标准：  
用户明确感知到  
即使换 UI 或换技术，这个能力依然成立

例如：  
衣物被结构化理解  
风格偏好被长期记住  
穿搭结果可被编辑和确认

产出 docs/product/capabilities.md  
每个能力包含：  
能力描述  
用户感知到的变化  
触发条件  
输出结果

禁止出现按钮名、接口名。

4）再用“能力”反推“产品流程”  
现在才进入路径层。

针对每个核心能力，回答：  
用户是通过怎样的连续行为触发它的？

这里用的是“产品流程”，而不是页面跳转。

每一条流程描述为：  
起点状态  
关键动作序列  
系统响应  
完成状态

产出 docs/product/flows.md

到这一步，一个人已经可以在脑中“跑一遍产品”，但还不知道 UI 长什么样。

5）最后才引入“页面与界面”，作为能力的载体  
 去看前端页面，但目的只有一个：  
这些页面各自承载了哪些能力和流程节点？

 为每一个“对用户有意义的页面”写一份说明，而不是全量页面。

每个页面文档固定结构：  
这个页面解决什么问题  
它承载了哪些能力  
用户在这里可以完成哪些关键动作  
这些动作分别对应哪条流程

产出 docs/product/screens/ 目录  
页面不是主角，能力和流程才是。

6）把 AI 当成“产品能力模块”，而不是技术模块  
关于 prompt，你要明确要求 AI 这样处理：

prompt 不属于代码，也不属于模型  
prompt 属于“产品决策层”

你让 AI 建立 docs/product/ai-capabilities.md  
每一个 AI 能力写清楚：  
它在产品中扮演什么角色  
用户什么时候会感知到它  
它需要哪些输入信息  
它会改变用户的什么决策

prompt 原文可以附在附录或单独文件，但正文只写“意图与效果”。

7）用“异常体验”而不是“错误处理”补完整度  
产品文档必须覆盖失败状态，但用产品语言。

 针对每条流程回答：  
如果这一步没成功，用户会经历什么？  
他会看到什么反馈？  
他还能不能继续往下走？

产出 docs/product/edge-cases.md

禁止出现报错码、异常栈，只写体验。

8）最后补一层“产品决策注解”，而不是工程 ADR  
不需要工程 ADR，需要的是“产品为什么这么设计”。

 写 docs/product/decisions.md  
每一条包含：  
当时的产品目标  
我们选择了什么方案  
放弃了什么方案  
这样做对用户意味着什么

这是你未来讲产品、融资、招聘的核心资产。

9）维护规则（产品版）  
 在 product/README.md 里加一段维护原则：

新增一个用户目标，必须补 goals 和 capabilities  
新增或改变用户路径，必须补 flows  
新增一个用户可感知页面，必须补 screens  
改变 AI 行为，必须补 ai-capabilities

明确一句话：  
代码可以先改，产品文档必须在同一迭代内补齐。
# 核心约束：
- 以『用户感知』为第一事实。
- 代码仅作为证据校验，禁止在文档中解释实现细节。
- 必须确保文档系统逻辑自洽。
