# Zara AI Coding 方法论

> 以下内容来自 Zara（文科生，无工程背景）在 AI 独立开发实践过程中的真实总结，横跨从零开始 Coding、工具选择、产品 Shipping 到心态建立的各个维度。代表作：ExcaliCore 白板录屏工具（基于 Excalidraw 开源项目，全程由 Zara + Claude Code 独立完成）。

---

## 一、Coding 不等于 Shipping：写代码只是开始

很多人以为 AI Coding 就是让 AI 帮你写代码，但真正的挑战是 Shipping——把产品真正交到用户手里。Vercel CEO 曾指出：Shipping 和 Coding 是两种不同的技能。写代码之外还有设计、测试、讲故事、教人使用、营销推广、迭代等等。实际上，Zara 在做产品的时候只有 30% 时间在写功能，70% 的时间在 Shipping 相关的事情。

**具体展开：**
- 给自己做产品 vs 给用户做产品，难度完全不是一个级别。给自己用，跑在本地就够了；给用户用，你需要部署、账号系统、支付、数据分析，还要把这一切都打通。
- 上线之后用户的使用环境千奇百怪，Chrome work、Edge 不 work，很多问题只有 ship 出去才会发现。
- 不碰代码、全程用提示词驱动 Claude Code 开发，绝大部分时间都在等输出和测试生成的结果。

---

## 二、站在巨人肩膀上：用开源工具拼乐高，别从零造轮子

初学者最容易犯的错误是从头开始建项目。更聪明的做法是找开源工具、现成的 API 和 SaaS 平台，在它们的基础上加入自己的功能，就像拼乐高而不是自己造积木。

**具体展开：**
- Zara 的录屏工具 ExcaliCore，就是基于开源白板软件 **Excalidraw** 开发的，白板核心功能直接复用，只需在上面加一个录屏逻辑即可，难度大大降低。
- 找开源软件 → 找各种 API → 找 SaaS 工具 → 在它们基础上连接和叠加，这就是 vibe coding 的乐高思维。

---

## 三、定义好 Scope：初学者先从简单做起

初学者最关键的一步不是选哪个工具，而是选一个难度合适的项目。范围定义太大，很难有正反馈，容易放弃。

**具体展开：**
- 先给自己做产品，而不是直接冲着用户需求去。给自己用的门槛低，功能范围小，更容易成功。
- 选择简单项目更容易产生正反馈，有了正反馈才有继续做下去的动力。
- 文科生 vibe coding 不是只能做 demo 了，有了 Claude Code 之后，即使完全不懂代码，也可以从头到尾做出可以上线的小应用。

---

## 四、Bug 解不掉？让 AI 上网查

当 AI 卡在同一个 bug 几轮都解不了时，不用一直追问它，换个思路：**让它去上网查别人是怎么解决的**。

**具体展开：**
- 指令参考："你去网上查一下别人是怎么解决这个问题的。"
- 大概率别人遇到过同样的问题，网上有解决方案，AI 查完再回来解决成功率会大幅提升。

---

## 五、先做 Playground，看到了才知道想要什么

设计这件事靠语言描述是说不清楚的，你得**看到效果**才知道自己想要什么。在让 AI 正式开始设计之前，先让它做一个 Playground，把它能实现的所有效果一口气展示出来，你再挑。这个技巧来自 Claude Code 负责人在活动上的亲身分享。

**具体展开：**
- **Animation Playground**：让 CC 把它能在前端实现的所有酷炫效果都放进一个 playground——打字特效（scramble）、渐变（gradients）、鼠标悬浮弹跳、3D 效果、数字动效等等，你先"逛一圈"，挑到喜欢的再说"按照这个来做"。
- **元素级 Playground**：想给某个按钮加特效？先让 CC 基于这个按钮做一个包含各种效果版本的 playground，玩一圈再挑，而不是靠文字描述"我想要一个有点动感的按钮"。
- **多版本对比**：同一个设计让它做多个版本（颜色、标题、布局各一套），你看着挑，不要依赖 AI 替你做审美决定。
- 提示词参考："给我做一个 playground，把你所有能实现的技巧都展示一下。"

## 六、把 AI 当老师：心理安全感 + Build First, Learn Later

AI 不只是你的工程师，更是你最好的老师——因为它给你提供任何人类老师都给不了的东西：**零评判的心理安全感**。而且它支持最适合成年人的学习方式：project-based learning，先做出东西，再回头问为什么。

**具体展开：**

*心理安全感：*
- 很多人不爱学习不是因为没有好奇心，而是怕问傻问题丢脸、怕被评判。那些你永远不好意思问人类程序员的"小白问题"，都可以问 AI。
- Claude 不会觉得你傻，不管什么问题都会耐心讲解，用类比帮你理解，还会夸你"问得真好"，没有抱怨、没有抵抗、没有情绪。
- AI 没有 ego，在 AI 面前你也没有 ego，学习就变成纯粹的好奇心驱动。
- 建议：多问傻问题，刨根问底，听不懂就让它再简化，用故事和类比再讲，直到懂为止。

*Build First, Learn Later（先做，后问）：*
- 成年人最有效的学习方式是 project-based learning：不是先上基础课，而是直接用 AI 编程工具动手做小产品，先让它 work、先得到正反馈，然后再问 AI："给我讲讲你怎么做出来的。"
- 学习要可持续，必须带来正反馈和快乐，先做出东西再理解原理，正反馈来得更快。
- 现在这个时代，只要你有主动性、好奇心和 AI，应该没有你学不会的东西。

*可视化理解代码库：*
- 让 CC 把代码库做成**分层架构图**：界面层（用户看到的）→ 逻辑层（hooks）→ 服务层（干活的引擎）→ 中央指挥部（app 文件）。
- 最有趣的玩法：让 CC 用"**群聊对话**"方式模拟各服务怎么通信——按下录制键，app 文件说"各位就位，开拍！"，各服务依次响应，像情景剧一样直观。
- 提示词参考："把我的代码库做成一个可交互式的可视化网页，面向没有编程经验的人，用聊天对话比喻各个服务之间怎么通信，所有术语都用大白话解释。"
- 比起 markdown 文档，更推荐做成**好看的前端页面**，更直观也更有趣。

## 七、Ship 一个产品的 SaaS 工具清单与成本

如果你想面向用户 ship 一个 Web 产品，这里是一个完整的工具选型参考（来自 ExcaliCore 的实战经验，共使用了 8 个系统）。

| 工具 | 职责 | 费用 |
|------|------|------|
| **Claude Code**（Pro/Max）| AI 编程，写所有代码 | $20/月起（Pro 版，重度使用建议升级）|
| **Vercel** | 网页部署与托管 | 轻量免费，商用建议付费 |
| **Supabase** | 数据库 + 用户账号体系 | 基础免费 |
| **Stripe** | 收款（收手续费，无订阅费）| 按交易收费 |
| **Namecheap** | 购买域名 | 约 $11/年（便宜词）|
| **PostHog** | 数据分析（用户量、使用时长）| 基本全免费 |
| **Google Sign In** | 账号体系 | 完全免费 |
| **Excalidraw**（开源）| 白板核心 | 免费 |

- 最低启动成本：一个 AI 编程工具（Claude Code Pro $20）+ 域名（约 $11）= **约 $31/月**，无商用需求时其他工具基本可免费。
- 正式商用后再升级 Vercel 和 Supabase 的付费版。

## 八、独立开发最爽的事：迭代节奏完全自主

不需要跟工程师沟通需求，不需要请示任何人，想迭代什么功能，跟 Claude 聊几轮就做好了。这种极快的迭代速度和自主感，是独立开发最独特的体验。

**具体展开：**
- 比如提词器透明度功能，就是"我跟 Claude 聊个两三轮它就做好了，就是几分钟的事"。
- 用户提反馈 → 当天就能上线修复。这种响应速度在传统开发模式下几乎不可能。

---

## 九、让 AI 采访你，而不是让你描述设计风格

用文字描述视觉风格对非设计师来说极其困难。聪明的做法是：**让 AI 先采访你**，通过出选择题来帮你确定想要的设计风格，而不是让你自己写一大段风格描述。

**具体展开：**
- 提示词参考："在开始设计之前，你先采访我一下，帮我确定我想要什么样的设计风格。"
- AI 会问：你希望用户来这里是什么感受？（能量高 / 平静 / 专业）你希望模仿哪些科技产品的风格？（Notion / Spotify / Apple）
- 你不需要懂"平静的 App 应该用什么颜色和字号"，你只需要回答这些选择题，剩下的 AI 自己翻译成具体设计语言。
- 关键洞察：选择题人人能回答，但从零写设计 brief 很难——把这个翻译工作交给 AI。

---

## 十、用 Huemint 选配色，把色值直接扔给 AI

配色和设计一样，你也得"看到了才知道想要什么"。[Huemint](https://huemint.com) 是一个配色 Playground，随机生成各种配色方案，找到喜欢的之后把色值复制给 AI，直接应用到产品里。

**具体展开：**
- 进入 Huemint，选择你需要的颜色数量（2色、3色、品牌色、渐变色、网站配色等），点击随机生成，找到顺眼的。
- 复制颜色代码（HEX）→ 粘贴给 CC → "把这套配色应用到我的产品里"。
- 比自己在脑海里想象颜色要高效且准确得多。

---

## 十一、用代码做 PPT：前端的可能性远超你想象

代码可以"画" PPT，而且可以做出静态工具完全无法实现的酷炫动态效果——hover 特效、3D 翻转、点击弹出、页面过渡、鼠标光标特效、嵌入实时前端交互……

**具体展开：**
- Zara 用 Claude Code 做出的 PPT，每一页都有过渡特效，可以嵌入前端交互组件，还有实时的鼠标光标特效。
- 小白一开始不知道"前端能干嘛"，这也是为什么先做 Playground 如此重要——你看到了才会想到用。
- 前端的表现力不是静态设计，而是一整套动态交互体验。

---

## 核心原则：Show, not tell

上面很多技巧（Playground、AI 采访你、Huemint 配色）背后有同一个哲学：**设计是视觉的，不是语言的。**

- 你跟 AI 沟通时，与其用文字描述，不如直接 show 给它看——参考图片、喜欢的网站截图、同类产品都是更好的输入。
- 同样，让 AI 跟你沟通时，也要求它用 demo、图片或可视化页面呈现，而不是一大段文字。
- Show not tell，双向都适用。
